## 一、 当前问题与核心痛点

### 1. 现状概述

我们在开发一个通过局域网、扫码即连的无线触控板与键盘项目。主控端（手机）使用纯原生前端构建了一套交互一体化的虚拟键盘，被控端（电脑）则基于 **Tauri (Rust)** 架构运行。

### 2. 核心痛点

* **沙箱隔阂与状态不可见：** 网页端（手机）与操作系统（OS）底层输入法（IME）存在天然屏障。手机无法得知电脑当前究竟是中文还是英文状态。

* **按键非标性：** 不同操作系统（Windows、macOS、Linux）切换中英文的物理按键/组合键各不相同（如 `Shift`、`Caps Lock`、`Ctrl + Space`），甚至同一系统下不同用户、不同输入法软件的习惯也不同。

* **状态异步（乱套问题）：** 如果没有初始校准，扫码连接后，手机端默认显示“中文”，而电脑实际处于“英文”状态，此时按下切换键，两边状态会互相同步翻转（手机变英、电脑变中），永远对不上。

---

## 二、 大体解决思路

为了追求极致的交互一体化，项目放弃了弹出手机自带输入法的折中方案，选择“完全模拟物理外设”的硬核路线：

1. **后端抹平平台差异：** 利用 Rust 的条件编译，针对 Windows、macOS、Linux 分别编写底层的输入法状态读取与按键模拟代码，但对外提供大统一的接口。

2. **双重校准机制：**

* **主动校准（刚连接时）：** 扫码连接成功的瞬间，Tauri 主动调用系统底层 API 获取当前输入法状态，并推给手机端，手机端据此初始化虚拟键盘 UI。

* **被动容错（使用中）：** 在手机端设计一个低成本的“手动反转/刷新”图标。一旦用户发现由于特殊软件干扰导致状态不同步，点击即可直接翻转手机端键盘“中/EN” 键的UI 状态，不发送任何按键，瞬间对齐。

3. **自定义快捷键兜底：** 被控端 GUI 面板允许用户自定义输入法切换快捷键，满足个性化需求。比如在被控端的GUI面板中添加一个配置：“我的电脑切换中英文的快捷键是：[ Shift ] 或 [ Ctrl + Space ]”，以便用户发现自己主控端键的“中/EN”键不起作用时自己去配置适合自己电脑的快捷键。

---

## 三、 项目结构整理（面向隔离开发）

为了让项目不沦为 `if/else` 的泥潭，我们在 Rust 后端采用 **Trait 抽象 + 条件编译（Conditional Compilation）** 的策略。核心思想是：**“前端一套代码，后端一个接口，编译时各走各路”**。

### 📂 推荐的项目目录结构

```text
src-tauri/src/

├── main.rs # Tauri 业务逻辑入口、注册 Command

├── platform/

│ ├── mod.rs # 核心：定义统一契约 (Trait) 与条件编译工厂

│ ├── windows.rs # Windows 专属：调用 Win32 API

│ ├── macos.rs # macOS 专属：调用 Carbon/Cocoa 框架

│ └── linux.rs # Linux 专属：通过 DBus 与 Fcitx/IBus 通信
```

### 🛠️ 核心代码分层设计

1. **在 `platform/mod.rs` 中定义统一契约（Trait）：**

```rust
pub trait PlatformHandler {

fn get_ime_status(&self) -> String; // 必须返回 "ZH" 或 "EN"

fn toggle_ime(&self, custom_keys: &Vec); // 模拟切换按键

}
```

2. **利用 `#[cfg]` 在 `mod.rs` 中做条件编译隔离：**

```rust
#[cfg(target_os = "windows")] mod windows;

#[cfg(target_os = "macos")] mod macos;

pub fn get_platform() -> Box {

#[cfg(target_os = "windows")] { Box::new(windows::WindowsHandler) }

#[cfg(target_os = "macos")] { Box::new(macos::MacosHandler) }

}
```

*这样在 Windows 上编译时，Mac 的代码和依赖压根不会进编译器，完美隔离。*

## 四、 核心攻坚：主控端键盘状态的“初始校准”

扫码进入页面的那一刻，如何让手机知道电脑现在的输入法状态？

### 1. 电脑端（Tauri）主动获取状态

当 WebSocket 连接建立成功，或者前端触发 `check_ime_status` 时，Tauri 根据当前平台执行底层代码：

* **Windows 落地：** 引入 `windows-sys` 库，调用 `GetForegroundWindow()` 获取当前聚焦的窗口，再通过 `ImmGetContext()` 和 `ImmGetConversionStatus()` 读取转换标志位。若 `(conversion & IME_CMODE_NATIVE) != 0` 则为中文，否则为英文。

* **macOS 落地：** 引入 `cocoa` 或 `objc` 库，通过 FFI 调用 Carbon 框架的 `TISCopyCurrentKeyboardInputSource`，获取当前输入法 Source ID。如果字符串内包含 `"pinyin"`、`"wubi"` 或 `"shuangpin"` 则判定为中文，否则为英文。

### 2. 状态传递与前端响应

1. Tauri 后端得出结果（例如 `"ZH"`）。

2. 通过局域网（WebSocket）向手机端发送初始化消息：`{"event": "ime_init", "status": "ZH"}`。

3. 手机端 JavaScript 收到消息，激活虚拟键盘上“中”字按键的高亮样式（若为 "EN" 则高亮“EN”）。

4. **至此，两端状态完美在起点对齐。**

---

## 五、 终点落地：按下“中/EN”键的实时调用

当两端状态对齐后，用户在手机上按下虚拟的“中/EN”键，其触发流程应该像一个真正的物理外设一样纯粹。

### 1. 触发流程流水线

1. **手机端（前端）：** 用户点击“中/EN”键。手机端不盲目切换自己的 UI，而是通过 WebSocket 向电脑发送一个纯粹的动作指令：`{"action": "press_ime_toggle"}`。

2. **电脑端（Tauri 业务层）：** 收到指令，调用大统一接口：

```rust
let handler = platform::get_platform();

handler.toggle_ime(&user_custom_keys);
```

3. **电脑端（底层物理模拟）：**

* 在 **Windows** 底层实现中：使用 `enigo` 或 Win32 `SendInput` 模拟按下并抬起 `LShift`（或用户在 GUI 里自定义的组合键）。

* 在 **macOS** 底层实现中：模拟按下并抬起 `Caps Lock`。

4. **状态闭环变更：**

* 电脑系统的输入法由于接收到了物理按键，自行切换了中英状态。

* 与此同时，手机前端在发送完指令后，**同步翻转自身的 UI 状态**（中文变英文，英文变中文）。

### 2. 交互兜底（防止意外脱节）

因为电脑上的用户可能会自己用物理键盘按了一下 `Shift` 导致状态脱节，你在手机端键盘设计的“长按'中/EN'键反转 UI 状态（不向电脑发指令）”**或者**“右上角小刷新按钮”就成了点睛之笔。它赋予了系统极高的容错率，用户体验直接拉满。

---

这套思路从**工程解耦（Trait+条件编译）**到**核心突破（底层API获取状态）**再到**体验闭环（物理模拟+手动兜底）**，逻辑非常闭环。现在，你可以按照这个步调，先从 Windows 版本的 `PlatformHandler` 开始着手实现了！
